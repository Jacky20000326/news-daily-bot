# Continue — 任務接續文件產生器

當使用者在較長的任務中需要中斷，呼叫此 skill 將當前對話的關鍵資訊整理成結構化文件，存放於專案中，以便開啟新 session 時能快速接續。

## 執行流程

### 步驟一：蒐集當前狀態

1. **回顧對話內容**：回顧整個對話，提取以下資訊：
   - 使用者最初的任務目標與需求
   - 已完成的工作項目
   - 尚未完成的工作項目
   - 重要的技術決策與原因
   - 遇到的問題與解決方式（或尚未解決的問題）
   - 任何使用者提供的偏好或約束條件

2. **檢查 git 狀態**：執行以下指令了解程式碼變更：

```bash
# 查看目前的分支與變更狀態
git status

# 查看未提交的變更摘要
git diff --stat

# 查看最近的提交紀錄（取得工作脈絡）
git log --oneline -5
```

3. **檢查是否已有接續文件**：

```bash
ls .claude/continue/ 2>/dev/null
```

如果已存在舊的接續文件，讀取其內容作為參考，後續會覆蓋更新。

### 步驟二：產生接續文件

將整理好的資訊寫入 `.claude/continue/SESSION.md`，使用以下格式：

```markdown
# 任務接續紀錄

> 產生時間：{YYYY-MM-DD HH:mm}
> 分支：{current branch}

## 任務目標

{使用者最初的需求描述，1-3 句話}

## 已完成項目

- [x] {已完成的工作項目 1}
- [x] {已完成的工作項目 2}
- ...

## 待完成項目

- [ ] {尚未完成的工作項目 1}
- [ ] {尚未完成的工作項目 2}
- ...

## 重要決策與脈絡

{記錄關鍵的技術決策、架構選擇、以及做出這些決策的原因。這些資訊對於接手的 session 至關重要。}

## 目前狀態

{描述程式碼目前的狀態：哪些檔案被修改了、是否有未提交的變更、是否有已知的 bug 或待修問題}

## 相關檔案

{列出本次任務中主要涉及的檔案路徑，方便新 session 快速定位}

- `path/to/file1.ts` — {簡短說明}
- `path/to/file2.ts` — {簡短說明}

## 下一步建議

{給新 session 的具體行動建議，依照優先順序排列}

1. {第一步該做什麼}
2. {第二步該做什麼}
3. ...
```

### 步驟三：確認與通知

1. 將文件寫入 `.claude/continue/SESSION.md`
2. 告知使用者文件已儲存，並簡述內容摘要
3. 提醒使用者：**在新 session 開啟時，只需告訴 Claude「請閱讀 `.claude/continue/SESSION.md` 接續之前的任務」即可**

## 注意事項

- 接續文件應該**自成一體**——新 session 的 Claude 不會有前次對話的上下文，所以文件中的描述必須足夠清晰完整
- 避免貼入大段程式碼，改用檔案路徑 + 行號引用
- 如果任務涉及敏感資訊（API key、密碼等），絕對不要寫入接續文件
- 每次呼叫 `/continue` 會覆蓋舊的 `SESSION.md`，確保始終是最新狀態
- `.claude/continue/` 目錄已加入 `.gitignore`，不會被提交到版本控制
